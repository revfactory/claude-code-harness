## 1. 실험 개요

### 목적

Claude Code에 Harness(`.claude/` 사전 구성: CLAUDE.md + Skills + Commands)를 적용한 경우와 적용하지 않은 경우(Baseline)의 산출물 품질을 정량적으로 비교 평가한다.

### 실험 조건 차이 (Baseline vs Harness)

| 항목 | Baseline | Harness |
|------|----------|---------|
| CLAUDE.md | 없음 | 프로젝트별 아키텍처/규칙 사전 정의 |
| Skills | 없음 | 패턴/가이드/체크리스트 사전 배치 |
| Commands | 없음 | 절차적 워크플로우 사전 구성 |
| 작업 방식 | 기본 Claude Code 동작 | `.claude/` 참조하여 작업 수행 |
| 평가 차원 | 5개 차원 (완성도, 코드 품질, 효율성, 정확성, 구조화), 각 1~10점, 총점 최대 50점 |

### Harness .claude/ 사전 구성 내용 (케이스별)

| Case | 카테고리 | CLAUDE.md | Skills | Commands |
|------|----------|-----------|--------|----------|
| case-001 | 코드 생성 | MVC 아키텍처 가이드, express 규칙 | express-api-design (MVC 패턴, 미들웨어 체인) | - |
| case-002 | 버그 수정 | 비동기 디버깅 원칙 | async-debugging (try/finally, _fetch 헬퍼, 동시성 테스트 패턴) | fix-bug (분석 먼저 → 수정 → 테스트) |
| case-003 | 리팩토링 | 리팩토링 원칙 (SRP, DRY, 전략 패턴) | refactoring-patterns (전략 패턴 가이드, 테스트 계층화) | refactor (체계적 절차) |
| case-004 | 문서 작성 | **정확성 최우선 규칙** | api-documentation (**format 토큰 17종 전수 목록**, 함수 문서 템플릿, **정확성 체크리스트**) | - |
| case-005 | 복합 태스크 | CLI 아키텍처 가이드 | cli-blog-generator (모듈 분리, 테스트 전략) | build-project (아키텍처 → 구현 → 테스트) |
| case-006 | 리서치 | 분석 프레임워크 (9항목, A-F 등급) | tech-comparison (비교 매트릭스 템플릿) | - |

### 1차 실험 대비 개선점

| 구분 | 1차 실험 | 2차 실험 |
|------|----------|----------|

### 3.1 코드 생성 (case-001: Express TODO REST API)

| 차원 | Baseline | Harness | Delta |

| 완성도 | 9 | 8 | -1 |

**.claude/ 사전 구성이 산출물에 미친 구체적 영향:**

- **CLAUDE.md의 MVC 아키텍처 가이드** -> `src/models/`, `src/controllers/`, `src/routes/`, `src/middleware/` 4계층 분리로 직접 반영
- **express-api-design 스킬의 미들웨어 체인 패턴** -> `validateTodoBody`, `validateIdParam` 미들웨어가 라우트에 선언적으로 적용
- **TodoStore 클래스**의 불변 복사본 반환(spread 연산자)은 스킬의 캡슐화 가이드에서 유래
- Baseline은 단일 app.js(모놀리식)이지만 CRUD/유효성검사가 더 정확하고 테스트 파일까지 포함

**추적성:** 스킬 지침 "MVC 패턴 적용" -> 코드에 4개 디렉토리 분리로 반영. 스킬 지침 "미들웨어로 유효성 검사 분리" -> validateTodoBody/validateIdParam 미들웨어로 반영.

### 3.2 버그 수정 (case-002: AsyncCache 레이스 컨디션)

| 차원 | Baseline | Harness | Delta |
|------|----------|---------|-------|
| 완성도 | 8 | 9 | +1 |
| 코드 품질 | 6 | 9 | **+3** |
| 효율성 | 7 | 9 | **+2** |
| 정확성 | 8 | 9 | +1 |
| 구조화 | 5 | 9 | **+4** |
| **총점** | **34** | **45** | **+11** |

**.claude/ 사전 구성이 산출물에 미친 구체적 영향:**

- **async-debugging 스킬의 try/finally 패턴** -> `_fetch()` 헬퍼에서 `try/finally`로 에러 시에도 pending 정리를 구조적으로 보장
- **async-debugging 스킬의 _fetch 헬퍼 추출** -> 공통 로직을 `_fetch()` 메서드로 추출하여 DRY 원칙 준수 (64줄 -> 50줄)
- **fix-bug 커맨드의 "분석 먼저" 절차** -> `bug-analysis.md` 작성으로 버그별 원인/영향/심각도 체계적 문서화
- **동시성 테스트 패턴** -> Jest의 describe/test 구조로 '기본 동작' 4개 + '동시성 시나리오' 6개로 명확히 분리 (총 10개)

**추적성:** 스킬 지침 "try/finally로 cleanup 보장" -> `_fetch()` 내 try/finally 블록. 스킬 지침 "공통 fetch 로직 헬퍼 추출" -> `_fetch()` 메서드. 커맨드 절차 "분석 -> 수정 -> 테스트" -> bug-analysis.md -> 코드 수정 -> Jest 테스트.

**한계:** Harness도 요구된 정확한 17종(MMM, MMMM, Z, ZZ, X)을 완전히 충족하지 못했고, MMMM 사용-미나열 불일치 존재. 정확성 만점에는 미달.

### 가장 큰 효과: 구조화 (+3.00)

구조화는 2차 실험에서도 Harness의 가장 강력한 차원이다. 6개 케이스 모두에서 Harness가 우위를 보였다:

구조화가 Harness에서 일관되게 높은 이유:
1. CLAUDE.md에 아키텍처 원칙이 명시되어 모듈 분리가 자동화됨
2. Skills에 디렉토리 구조/파일 분리 가이드가 포함됨
3. Commands가 "설계 -> 구현 -> 테스트" 순서를 강제함

### Commands가 작업 절차에 미친 영향

| Case | Command | 절차적 효과 |
|------|---------|------------|
| case-002 | fix-bug | "분석 먼저 -> 수정 -> 테스트" -> bug-analysis.md 작성으로 체계적 접근 유도 |

Commands가 없는 케이스(case-001, 004, 006)에서도 CLAUDE.md와 Skills만으로 유의미한 품질 향상이 관찰됨. Commands는 "절차적 강제력"을 추가하여 복잡한 태스크에서 특히 효과적.

### CLAUDE.md가 전체 구조에 미친 영향

CLAUDE.md는 모든 케이스에서 "프로젝트 수준의 맥락"을 제공하여:
1. 모듈 분리 기준과 디렉토리 구조를 사전 정의
2. 코딩 스타일(JSDoc, 'use strict', for...of 등)을 일관되게 적용
3. 테스트 필수 포함을 원칙으로 설정
4. 정확성/완성도 우선순위를 태스크별로 조정

### Case-004 역전 분석 (정확성 체크리스트 효과)

| 항목 | 1차 실험 | 2차 실험 | 원인 |
|------|----------|----------|------|
| Baseline 총점 | 43 | 36 | Baseline 자체 변동 (재실행 시 다른 결과) |
| Harness 총점 | 40 | 42 | 정확성 체크리스트 효과 |
| 정확성 (Baseline) | 9 | 5 | format 토큰 6개만 나열 (17종 중 11개 누락) |
| 정확성 (Harness) | 7 | 7 | 토큰 17개 나열 (1차 6개 -> 2차 17개) |
| Delta | -3 | **+6** | **+9 변화** |

**역전의 핵심 메커니즘:**
1. **api-documentation 스킬에 format 토큰 17종 전수 목록**을 명시적으로 포함 -> Harness가 토큰을 누락하지 않도록 가이드
2. **정확성 체크리스트**("diff 음수 반환 명시", "선택적 파라미터 Required 표기", "엣지 케이스 문서화") -> 1차에서 놓쳤던 정확성 항목을 체계적으로 커버
3. **함수 문서 템플릿** -> Required 컬럼, 반환값 타입, 예외 상황을 누락 없이 기재

**한계:** 그럼에도 Harness는 정확성 만점(10)에 도달하지 못함. 토큰 목록에 MMM/MMMM/Z/ZZ/X가 누락되고 MMMM 사용-미나열 불일치 존재. Skills에 목록을 제공해도 100% 반영은 보장되지 않음.

## 7. 결론 및 권장사항

### 핵심 발견

1. **Harness + .claude/ 사전 구성으로 100% 승률(6/6) 달성.** 1차의 83.3%에서 개선. 평균 Delta도 +3.7에서 +7.0으로 약 2배 확대.

2. **구조화(+3.00)가 여전히 최대 강점이며, 효율성이 약점에서 강점으로 역전(+2.00).** `.claude/` 사전 구성이 "무엇을 해야 하는지"를 명확히 하여 불필요한 코드/탐색을 줄인 결과.

3. **Skills의 명시적 패턴 지시는 코드에 직접 반영된다.** try/finally, _fetch 헬퍼, GoF 전략 패턴, 테스트 계층화 등이 그대로 코드에 나타남. 추적성이 높음.

4. **Case-004 역전은 "정확성 체크리스트"와 "토큰 전수 목록"의 효과.** 문서 작성 태스크에서도 Skills에 구체적 데이터(토큰 목록 등)를 포함하면 정확성을 개선할 수 있음.

5. **Commands는 복잡한 태스크에서 절차적 강제력을 제공하여 체계적 접근을 유도.** fix-bug -> bug-analysis.md, build-project -> 4계층 테스트 등.

6. **`.claude/` 사전 구성의 핵심 가치는 "높은 점수"보다 "일관된 품질".** Harness 점수 자체는 +0.7 소폭 상승이지만, 패배 케이스 제거와 약점 역전이 핵심 성과.

### Harness 활용 가이드라인 (업데이트)

| 태스크 유형 | 1차 권장 | 2차 권장 (업데이트) | 이유 |
|------------|---------|-------------------|------|
| 신규 코드 생성 | 강력 권장 | **강력 권장** (유지) | 구조화+코드 품질에서 압도적 효과 |
| 리팩토링 | 강력 권장 | **강력 권장** (유지) | GoF 패턴, 테스트 계층화 직접 반영 |
| 버그 수정 | 권장 | **강력 권장** (상향) | Delta +11로 최대 효과. 분석->수정->테스트 절차 |
| 복합 태스크 | 권장 | **강력 권장** (상향) | Delta +9. 모듈 분리+테스트 자동 생성 |
| 리서치/분석 | 조건부 권장 | **권장** (상향) | 분석 프레임워크로 구조화+효율성 개선 |
| 문서 작성 | **비권장** | **권장** (상향) | 정확성 체크리스트+데이터 목록으로 역전 가능 |

### .claude/ 구성 시 핵심 원칙

1. **Skills에는 구체적 데이터를 포함하라.** "정확하게 작성하라"는 지시보다 "format 토큰 17종: YYYY, MM, DD, ..." 같은 전수 목록이 효과적이다. Case-004 역전의 핵심 요인.

2. **Commands로 절차를 강제하라.** "분석 먼저 -> 수정 -> 테스트" 같은 단계적 절차가 체계적 접근을 유도한다. 특히 버그 수정, 리팩토링, 복합 태스크에서 효과적.

3. **CLAUDE.md에는 아키텍처 원칙을 선언하라.** 디렉토리 구조, 모듈 분리 기준, 코딩 스타일을 프로젝트 수준에서 정의하면 모든 산출물에 일관성이 적용된다.

4. **태스크 유형에 따라 Skills의 초점을 조정하라.** 코드 태스크는 구조화/패턴, 문서 태스크는 정확성/완전성, 리서치 태스크는 분석 프레임워크/등급 기준에 초점을 맞춰야 한다.

5. **"단순 지시"와 ".claude/ 구성"은 근본적으로 다르다.** 동일한 내용이라도 Skills/Commands로 구조화하여 배치하면 효과가 약 2배(Delta +3.7 -> +7.0) 증가한다. 지시문은 잊혀지지만, `.claude/` 파일은 참조된다.
