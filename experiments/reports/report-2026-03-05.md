## 1. 실험 개요

### 목적
Claude Code의 기본 동작(Baseline)과 Harness(구조화된 프롬프트/워크플로우 강화) 적용 시 산출물 품질 차이를 정량적으로 비교 평가한다.

### 실험 조건
| 항목 | 설명 |
|------|------|
| 비교 대상 | **Baseline** (기본 Claude Code) vs **Harness** (강화된 워크플로우) |
| 케이스 수 | 6개 |
| 카테고리 | 코드 생성, 버그 수정, 리팩토링, 문서 작성, 복합 태스크, 리서치 |
| 평가 차원 | 완성도(completeness), 코드 품질(quality), 효율성(efficiency), 정확성(correctness/accuracy), 구조화(architecture/structure) |
| 점수 범위 | 각 차원 1~10점, 총점 최대 50점 |

### 평가 케이스 목록

| Case | 카테고리 | 제목 |
|------|----------|------|
| case-001 | 코드 생성 | REST API 서버 구현 |

| case-003 | 리팩토링 | 스파게티 calc() 함수 리팩토링 |
| case-004 | 문서 작성 | datekit 라이브러리 README.md 작성 |
| case-005 | 복합 태스크 | CLI 기반 마크다운 블로그 생성기 |
| case-006 | 리서치 | React 상태 관리 라이브러리 5개 비교 분석 |

## 3. 카테고리별 분석

### 3.1 코드 생성 (case-001: REST API 서버 구현)

| 차원 | Baseline | Harness | Delta |
|------|----------|---------|-------|
| 완성도 | 8 | 9 | +1 |
| 코드 품질 | 6 | 9 | **+3** |
| 효율성 | 8 | 7 | -1 |
| 정확성 | 8 | 9 | +1 |
| 구조화 | 5 | 9 | **+4** |
| **총점** | **35** | **43** | **+8** |

**핵심 발견사항:**
- Harness의 가장 큰 강점은 **구조화(+4)**와 **코드 품질(+3)**. MVC 패턴 기반으로 models/controllers/routes/middleware를 체계적으로 분리했다.
- Baseline은 단일 파일(app.js)에 모든 로직을 집중하여 간결하나 확장성이 부족하다.
- Harness는 ID 파라미터 검증(정수/양수), JSON 파싱 에러 처리 등 엣지 케이스를 더 충실히 다루었다.
- 효율성에서는 Baseline이 2개 파일(테스트 포함)로 간결하게 구현하여 소폭 우위(-1).

### 3.2 버그 수정 (case-002: 비동기 레이스 컨디션)

| 차원 | Baseline | Harness | Delta |
|------|----------|---------|-------|
| 완성도 | 9 | 10 | +1 |
| 코드 품질 | 7 | 9 | **+2** |
| 효율성 | 8 | 7 | -1 |
| 정확성 | 8 | 9 | +1 |
| 구조화 | 7 | 9 | **+2** |
| **총점** | **39** | **44** | **+5** |

**핵심 발견사항:**
- 양쪽 모두 3가지 레이스 컨디션 버그를 올바르게 수정했다.
- Harness는 `try/finally` 패턴과 `_fetch` 헬퍼로 에러 시 pending 정리를 더 견고하게 보장한다.
- Harness는 12개 테스트(36+ assertion) vs Baseline 8개 테스트로 더 넓은 커버리지를 제공한다.
- Harness는 bug-analysis.txt와 buggy-cache.js로 버그 분석 문서와 원본 코드를 보존하여 완성도 만점(10).
- 효율성에서는 Baseline이 61줄로 간결하게 구현한 반면, Harness는 107줄(-1).

- 테스트 규모: Harness 148개 어설션 vs Baseline 30개 테스트.
- 효율성에서는 Baseline의 단일 파일 접근이 이 규모에서 오히려 적절(-1).

## 4. 차원별 심층 분석

### 차원별 평균 점수 비교

| 차원 | Baseline 평균 | Harness 평균 | 평균 Delta |
|------|---------------|--------------|------------|
| 완성도 | 8.67 | 9.17 | **+0.50** |
| 코드 품질 | 7.50 | 8.67 | **+1.17** |
| 효율성 | 8.00 | 7.50 | **-0.50** |
| 정확성 | 8.33 | 8.50 | **+0.17** |
| 구조화 | 6.67 | 9.00 | **+2.33** |

### Harness 효과가 큰 차원 (내림차순)

1. **구조화 (+2.33)** — Harness의 가장 일관된 강점. 6개 중 5개 케이스에서 Harness가 우위 또는 동등. 모듈 분리, 관심사 분리, 계층 구조가 체계적으로 적용된다.
2. **코드 품질 (+1.17)** — 6개 중 4개 케이스에서 Harness가 우위. JSDoc 문서화, 디자인 패턴 적용, 테스트 포함이 일관적이다.
3. **완성도 (+0.50)** — 6개 중 3개 케이스에서 Harness가 우위, 3개에서 동등. 엣지 케이스 처리, 문서화, 유틸리티 메서드 추가 등으로 완성도가 더 높다.

### Harness 효과가 작거나 역전되는 차원

4. **정확성 (+0.17)** — 거의 동등. 코드 관련 태스크에서는 Harness가 소폭 우위이나, 문서 작성(case-004)에서 정확성이 크게 뒤처짐(-2).
5. **효율성 (-0.50)** — **유일하게 Baseline이 우위인 차원.** 6개 중 3개 케이스에서 Baseline이 우위, 2개에서 동등, 1개에서 Harness 우위. Harness의 체계적 구조는 파일/코드량 증가를 수반한다.

### 패턴 분석

- **구조화 + 코드 품질의 동반 향상**: 이 두 차원은 모든 케이스에서 같은 방향으로 움직인다. Harness가 모듈 분리를 하면 자연스럽게 코드 품질도 올라가는 구조적 연관성이 있다.
- **효율성과 구조화의 역상관**: 구조화 delta가 클수록 효율성 delta가 작거나 음수인 경향이 있다 (case-001: 구조+4/효율-1, case-003: 구조+4/효율-1).
- **정확성의 카테고리 의존성**: 코딩 태스크에서는 Harness가 정확성에서 우위이나, 문서/리서치에서는 일관되지 않다.

## 5. 트레이드오프 분석

### Harness가 효과적인 경우

| 상황 | 이유 | 대표 케이스 |
|------|------|-------------|
| **코드 생성** (신규 구현) | MVC 등 아키텍처 패턴을 자동 적용, 모듈 분리가 큰 가치를 발휘 | case-001 (+8) |
| **리팩토링** | 관심사 분리, 디자인 패턴 적용, 파일 분리가 리팩토링의 핵심 가치와 일치 | case-003 (+6) |
| **버그 수정** | 더 견고한 패턴(try/finally), 넓은 테스트 커버리지, 버그 분석 문서화 | case-002 (+5) |
| **복합 태스크** | 다수 기능을 모듈화하여 관리, 테스트 코드 자동 생성 | case-005 (+4) |

**공통 패턴:** Harness는 "코드를 작성하는" 태스크에서 구조적 우수성과 코드 품질 향상을 통해 일관되게 높은 효과를 보인다.

### Harness가 비효율적인 경우

| 상황 | 이유 | 대표 케이스 |
|------|------|-------------|
| **문서 작성** | 코드 구조화 능력이 문서 정확성으로 전이되지 않음. 오히려 한국어 표기, 불완전한 토큰 목록 등 정확성 하락 | case-004 (-3) |

**Case-004 심층 분석:**
- Harness는 구조화(+1)에서만 우위를 보였고, 코드 품질(-1), 효율성(-1), 정확성(-2)에서 뒤처졌다.
- format 토큰을 16개 중 6개만 문서화하여 불완전하고, relative 함수의 선택적 파라미터를 누락했다.
- 상대 시간 출력을 한국어('1분 전')로 표기하여 실제 라이브러리 동작과 불일치 가능성이 있다.
- **해석:** Harness의 구조화 우선 접근은 "코드를 얼마나 잘 짜는가"에는 효과적이나, "정보를 얼마나 정확하게 전달하는가"에는 오히려 방해가 될 수 있다. 문서 작성은 구조보다 정확성과 완전성이 핵심이다.

### 효율성 차원의 트레이드오프

| Case | 효율성 Delta | 구조화 Delta | 분석 |
|------|-------------|-------------|------|
| case-001 | -1 | +4 | 2파일 → 6파일. 소규모 프로젝트에 과한 분리 |
| case-002 | -1 | +2 | 61줄 → 107줄. 헬퍼/유틸리티 추가 |
| case-003 | -1 | +4 | 1파일 → 3파일. 클래스 기반 패턴 오버헤드 |
| case-004 | -1 | +1 | 불필요한 상세 구조가 정확성을 희생 |
| case-005 | 0 | +2 | 모듈 분리가 적절한 규모의 프로젝트 |
| case-006 | +1 | +1 | 리서치에서는 정보 효율성이 구조와 양립 |

**핵심 트레이드오프:** Harness는 **구조화를 위해 효율성을 소폭 희생**하는 경향이 있다. 평균적으로 효율성 -0.50점을 잃고 구조화 +2.33점을 얻는다. 이 트레이드오프는 프로젝트 규모가 클수록 유리하게 작용한다(구조화의 가치가 커지므로).

## 6. 결론 및 권장사항

### 핵심 발견

1. **Harness는 83.3% 승률(5/6)로 전반적으로 우수하다.** 평균 총점 42.8 vs 39.2로 3.7점(7.4%) 차이.

2. **구조화(+2.33)와 코드 품질(+1.17)이 Harness의 핵심 강점이다.** 모듈 분리, 디자인 패턴 적용, 테스트 커버리지에서 일관된 우위를 보인다.

3. **효율성(-0.50)은 Harness의 유일한 약점이다.** 체계적 구조는 필연적으로 파일 수와 코드량을 증가시킨다. 다만 이 트레이드오프는 프로젝트 규모가 커질수록 유리하다.

4. **문서 작성 태스크에서는 Harness가 오히려 불리하다.** 구조화 능력이 정보 정확성으로 전이되지 않으며, 불완전한 정보 기재와 실제 동작 불일치 문제가 발생한다.

5. **코드 관련 태스크에서 Harness의 효과가 극대화된다.** 코드 생성(+8), 리팩토링(+6), 버그 수정(+5) 순으로 효과가 크다.

### Harness 활용 가이드라인

| 태스크 유형 | 권장 | 이유 |
|------------|------|------|
| 신규 코드 생성 | **강력 권장** | 아키텍처 설계, 모듈 분리, 패턴 적용에서 큰 효과 |
| 리팩토링 | **강력 권장** | 관심사 분리, 파일 구조 개선이 핵심 가치와 일치 |
| 버그 수정 | **권장** | 견고한 패턴, 넓은 테스트, 분석 문서화 |
| 복합 태스크 | **권장** | 다기능 모듈화, 테스트 자동 생성 |
| 리서치/분석 | **조건부 권장** | 최신 정보 활용과 구조화에서 소폭 우위, 코드 품질은 Baseline 우위 가능 |
| 문서 작성 | **비권장** | 정확성 하락 위험. Baseline이 API 정밀성에서 우수 |

### 향후 개선 방향

1. **효율성 최적화:** 프로젝트 규모에 따른 적응적 구조화 수준 도입. 소규모 프로젝트에서는 과도한 파일 분리를 억제하는 휴리스틱 필요.

2. **문서 작성 정확성 강화:** 문서 태스크에서 API 소스코드와의 크로스 체크 메커니즘 도입. format 토큰, 함수 시그니처 등의 완전성 검증 단계 추가.

3. **테스트 일관성:** Harness는 case-001에서 테스트 미포함, Baseline은 case-005에서 테스트 미포함. 모든 코딩 태스크에서 테스트를 필수 산출물로 포함하도록 개선 필요.

4. **카테고리 적응 전략:** 태스크 카테고리에 따라 Harness의 강조점을 자동 조정하는 메커니즘 검토. 코드 태스크 → 구조화 강조, 문서 태스크 → 정확성 강조.
