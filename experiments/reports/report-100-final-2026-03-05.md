## 6. Harness 효과 메커니즘

### 6.1 CLAUDE.md → 아키텍처 일관성 (+4.1)

프로젝트 구조와 모듈 분리 규칙을 사전 정의함으로써 Baseline의 "단일 파일 모놀리식" 경향을 방지. Baseline의 아키텍처 평균이 4.7인 반면, Harness는 8.8을 달성.

### 6.2 Skills → 패턴/알고리즘 직접 전이 (+2.4 코드 품질, +1.3 정확성)

| Skill | 전이된 패턴 | 코드 품질 효과 | 정확성 효과 |
|-------|-------------|----------------|-------------|
| express-api-design | MVC, 미들웨어 체인 | TodoStore 캡슐화 | — |
| async-debugging | _fetch() 헬퍼, try/finally | DRY, 에러 안전 | 타이밍 이슈 차단 |
| refactoring-patterns | GoF 전략 패턴, 팩토리 | OCP 준수 | 경계값 테스트 |
| interpreter-design | Pratt Parser, AST 팩토리 | 10파일 분리 | 연산자 우선순위 |
| microservice-patterns | SAGA 범용 설계 | 에러 계층 | 보상 트랜잭션 |
| query-engine-design | 3단계 파이프라인 | Planner 계층 | **SQL 3값 NULL** |
| crdt-algorithms | RGA, 인과성 버퍼 | O(1) 자료구조 | **퍼즈 테스트** |

### 6.3 Commands → 절차적 품질 보장 (+2.5 문서화, +1.9 테스트)

- `fix-bug`: "분석 먼저, 수정 후" → bug-analysis.md 생성
- `refactor`: 4계층 테스트 체계화 (통합/단위/SRP/경계값)
- `generate`: 모듈 분리 + 테스트 + package.json 포함

### 6.4 복합 효과: 차원 간 시너지

```
CLAUDE.md(아키텍처 규칙)
    ↓ 모듈 분리 강제
Skills(패턴 가이드)     → 코드 품질 + 정확성 향상
    ↓ 테스트 전략 포함
Commands(절차 강제)     → 테스트 커버리지 + 문서화 향상
    ↓ package.json 생성
                        → 개발 환경 향상
```

### 핵심 발견

> **1. Harness는 Baseline 대비 평균 22.4점(39%) 우위**
>
> 100점 체계에서 Baseline 57.4점 vs Harness 79.8점. 10차원 확장으로 기존 5차원에서 숨겨져 있던 테스트(+1.9), 에러 처리(+1.9), 문서화(+2.5), 개발 환경(+2.5) 격차가 명시적으로 드러남.

> **2. 복합 시스템에서 Harness 효과 극대화 (Delta +28.0)**
>
> 인터프리터, 마이크로서비스, SQL 엔진, CRDT 에디터 같은 고난이도 복합 시스템에서 아키텍처 설계가 전체 품질을 결정. Baseline은 단일 파일 모놀리식으로 수렴하는 반면, Harness는 교과서적 모듈 분리를 일관되게 달성.

> **3. 아키텍처와 확장성이 가장 큰 격차 (각 +4.1, +3.3)**
>
> Baseline의 아키텍처 평균 4.7점(단일파일 수준)은 근본적 한계. Harness의 8.8점은 사전 구조 가이드라인의 직접적 효과.

> **4. Baseline의 구조적 약점: 개발 환경 3.9점**
>
> package.json, 실행 스크립트, 테스트 프레임워크 설정 등 프로젝트 인프라가 체계적으로 부재. Harness는 Commands가 이를 강제하여 6.4점 달성.

### 최종 메시지

> **Harness(.claude/ 사전 구성)는 Claude Code의 산출물 품질을 57점에서 80점 수준으로 끌어올리며, 특히 아키텍처(+4.1), 확장성(+3.3), 문서화(+2.5) 차원에서 결정적 차이를 만든다. 복잡한 시스템일수록 그 효과는 더욱 커진다.**
